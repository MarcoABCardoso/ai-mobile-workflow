# ADR 005 — Fastify as the backend service framework

**Status:** Accepted  
**Date:** 2026-06-15

## Context

Every backend service in this workflow needs an HTTP framework. The choice locks in the routing style, typing approach, and—critically—how the OpenAPI specification is produced.

## Decision

**Fastify** with `@fastify/swagger` (spec generation) and `@fastify/swagger-ui` (docs UI).

## Reasons

- **Well-established.** Fastify has been production-stable since 2018 and is actively maintained by a large contributor base.
- **TypeScript-native.** Route schemas are defined in TypeScript and enforced at compile time; no separate type-generation step needed for request/response shapes.
- **OpenAPI spec is generated from code.** `@fastify/swagger` derives the OpenAPI spec from the route schema definitions. This makes the spec an accurate reflection of the implementation rather than a hand-written document that can drift. The generated spec is then used as the source of truth for the shared API client (see ADR 007).
- **Performance.** Consistently benchmarks among the fastest Node.js frameworks. Not the reason it was chosen, but a useful property.
- **Auth plugin pattern.** `@fastify/jwt` or a custom `fastify.addHook('preHandler', ...)` integrates cleanly with the JWT validation required by ADR 004 providers.

## What this means in practice

- Services start with a Fastify app defined in `src/index.ts`.
- Every route declares a `schema` object (request body, params, response) using JSON Schema. No schema → no route merges.
- The OpenAPI spec is regenerated by running `npm run generate:openapi` in the service directory after any route change. This script starts the app, calls `app.swagger()`, and writes `openapi.yaml`.
- `openapi.yaml` is a **build artifact**, not a hand-written source file. It lives in the service directory but is always regenerated from code—never edited directly.
- `@fastify/swagger-ui` serves interactive docs at `/docs` in development.

## Trade-offs

- Less ecosystem breadth than Express (fewer third-party plugins), but coverage of needed features (auth, CORS, rate-limiting, multipart) is complete.
- JSON Schema for route definitions is more verbose than Zod or TypeBox at small scale; this is acceptable given the auto-generated OpenAPI benefit.

## Alternatives considered

- **Express** — larger ecosystem but no TypeScript-first story and no OpenAPI generation without additional libraries.
- **Hono** — excellent TypeScript story and runs on multiple runtimes, but newer and less field-tested at the time of this decision.
- **NestJS** — full framework with built-in OpenAPI; ruled out because its decorator-heavy style generates verbose scaffolding that is harder for AI to produce correctly and review.
