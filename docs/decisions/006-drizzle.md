# ADR 006 — Drizzle ORM for database access

**Status:** Accepted  
**Date:** 2026-06-15

## Context

Services that persist data need a way to talk to PostgreSQL (the default database — see ADR 005 context). The ORM choice affects type safety, migration strategy, cold-start performance, and how readable the generated SQL is.

## Decision

**Drizzle ORM** with `drizzle-kit` for schema migrations. PostgreSQL via `pg` (node-postgres) as the driver.

## Reasons

- **No binary engine.** Unlike Prisma, Drizzle ships no Rust query engine binary. This matters for the scale-to-zero setup (Cloud Run / Container Apps): there is no engine warm-up on cold start.
- **SQL-first.** Drizzle generates readable, predictable SQL. When an AI writes or reviews a query, the SQL is visible and auditable — not hidden behind a DSL that compiles to something opaque.
- **TypeScript schema.** Tables are defined in `.ts` files using Drizzle's table builders. No separate `.prisma` schema language means one fewer file format in the project, and the schema participates in the normal TypeScript compiler checks.
- **`drizzle-kit` migrations.** Schema diffing and migration generation are handled by `drizzle-kit generate` and `drizzle-kit migrate`. Migrations are plain SQL files in `drizzle/` — reviewable, version-controlled, and runnable without the ORM at hand.
- **Stable.** Drizzle ORM reached v1.0 and is in widespread production use. It is not a new experiment.

## What this means in practice

- Each service that needs a database has:
  - `src/db/schema.ts` — table definitions (the source of truth for the data model)
  - `src/db/index.ts` — Drizzle client instance, exported for use in route handlers
  - `drizzle/` — generated SQL migration files (committed, never edited by hand)
  - `drizzle.config.ts` — points drizzle-kit at the schema and migration output directory
- `DATABASE_URL` is the only required env var for database connectivity. It is injected from the secrets store at runtime.
- Convention: snake_case table and column names; UUID primary keys generated by the database (`defaultRandom()`).
- The AI must run `npx drizzle-kit generate` after any schema change and commit the resulting migration file before raising a PR.

## Trade-offs

- Drizzle has less ecosystem tooling (GUI clients, etc.) than Prisma. The migration SQL files and `drizzle-studio` cover the common needs.
- Drizzle does not generate types from an existing database schema (introspection is limited). Greenfield projects only is a constraint this workflow already imposes (ADR scope), so this is not a practical limitation.

## Alternatives considered

- **Prisma** — most widely used ORM in the Node.js/TS ecosystem. Ruled out because the Rust query engine adds cold-start latency in the scale-to-zero dev and prod environment this workflow targets.
- **Raw `pg` / `postgres.js`** — maximum control, no abstraction overhead. Ruled out because it requires manual type mapping and offers no migration tooling.
- **TypeORM** — established but relies on decorators (same complexity concern as NestJS) and has had reliability issues with complex join queries.
